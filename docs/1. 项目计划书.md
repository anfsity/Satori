# Satori 项目计划书

## 一、项目概述

### 1. 项目背景
随着互联网文化的快速发展，网络流行语、中文梗（Meme）以及特定圈层的黑话层出不穷，迭代速度极快。对于普通网民或语言研究者而言，理解这些新兴网络词汇的含义和演变过程具有一定的门槛。当前常规搜索引擎往往难以精确捕捉这些词汇的特定语境和幽默内核。因此，本项目旨在开发一个名为 Satori 的中文梗与黑话语义搜索引擎，为用户提供精准、高效的网络流行语查询服务，打破互联网语言壁垒，促进网络文化的跨圈层交流。

### 2. 项目目标
本项目旨在构建一个专注于中文网络结构化语义检索的后端系统。核心目标包括：
- 实现基于向量检索的语义匹配，使用户可以通过自然语言描述查找到相关的梗或黑话。
- 搭建高性能、易扩展的后端检索服务。
- 构建从原始语料到向量数据库的自动化数据处理管道。

### 3. 开发环境与条件
- **开发语言**：Rust（后端服务、数据处理组件）
- **Web 框架**：Axum
- **数据库**：LanceDB（向量数据库，用于存储和检索词频与嵌入信息）
- **模型推理引擎**：Candle（基于 Rust 的轻量级深度学习框架）
- **分词工具**：HuggingFace Tokenizers
- **嵌入模型**：bge-small-zh-v1.5
- **运行环境**：Linux/Unix 系统
- **版本控制与协作**：Git & GitHub

## 二、需求分析

### 1. 目标用户与使用场景
- **目标用户**：对互联网流行语感兴趣的普通网民、泛二次元文化爱好者、语言及社会学研究者。
- **使用场景**：
  - **白话查黑话**：用户输入一句普通表达，系统返回意思接近的黑话或网络梗。
  - **黑话查解释**：输入不懂的黑话，查到它对应的正常说法、解释和例句。
  - **模糊语义搜索**：用户通过输入模糊的意图或场景描述，检索到相关的黑话词条。

### 2. 核心功能清单
- **自然语言查询解析**：接收用户的自然语言输入并进行标准化处理。
- **文本向量化（Embedding）**：利用本地部署的模型服务将查询文本转化为高维向量。
- **语义相似度检索**：在向量数据库中进行近似最近邻（ANN）查找，并返回匹配最相关的数据条目。
- **数据导入机制**：解析并处理原始文本或 JSON 数据集，构建向量索引。

### 3. 非功能需求
- **性能要求**：以检索系统的稳定与低延迟为核心指标，接口响应时间在合理范围内。
- **安全要求**：服务健壮，防范恶意的大规模查询；通过合理的限流与接口保护机制保障系统稳定。
- **兼容性要求**：提供标准的 RESTful API 接口，便于未来前端独立开发或接入各类客户端请求。

## 三、技术路线

### 1. 总体架构
Satori 项目采用模块化架构进行开发，后端主逻辑与数据处理相分离。系统主要由三个阶段（或组件）构成：
- **数据索引流水线（Indexer）**：负责读取语料库，通过分词和嵌入模型提取文本特征，将结果与元数据共同写入向量数据库。
- **推理与计算引擎（Core）**：封装模型加载、向量生成等核心算法逻辑。
- **搜索引擎服务（API）**：基于 Axum 框架构建 HTTP 服务，解析客户端请求，连接向量数据库实施查询并返回检索结果。

### 2. 主要技术
- **后端架构**：全程采用 Rust 语言，保证系统在处理计算密集型任务（如词法解析与内存安全）和高并发时的资源利用效率。
- **搜索策略**：采用密集向量检索技术，相较于传统的倒排索引（TF-IDF或BM25），能在语义层面更精准地匹配用户意图。
- **模型推理**：摒弃沉重的完整深度学习框架，采用 Candle 搭配轻量级基础模型，实现本地部署和极低开销推理。

### 3. 数据设计初步考虑
- **主要实体或对象**：梗（Card）。其中包含梗的唯一标识、标题、描述、来源、热度等标签。
- **关键数据结构**：
  - 文档存储：JSON / 结构化记录格式存储原始信息。
  - 向量存储：结合 LanceDB 实现混合检索，内部维护词条 ID、关联的文本向量数组及若干元数据过滤字段（如：分类、日期）。

## 四、进度安排

### 1. 阶段划分
1. **需求调研与设计（阶段一）**：确定模型及数据库选型，搭建 Rust 基础项目骨架，设计数据流。
2. **数据处理与开发（阶段二）**：收集外部词库数据并清洗，搭建 Axum 后端 API，实现从语料导入到向量检索的完整链路。
3. **测试优化与交付（阶段三）**：编写冒烟测试与回归测试，优化模型检索效果与接口性能，完成最终文档。

### 2. 时间进度表
- 第 1 周：阶段一（选型验证、架构设计与基础环境搭建）。
- 第 2 周：阶段二（清洗数据、实现核心搜索接口与组件开发）。
- 第 3 周：阶段三（联调优化、自动化测试与项目交付）。

### 3. 风险与应对
- **模型效果不佳**：轻量级模型 `bge-small-zh-v1.5` 在垂直领域或部分罕见黑话上可能存在泛化性不足。**应对思路**：如有必要，在后期考虑使用微调模型。
- **本地推理性能瓶颈**：CPU 推理文本向量耗时可能无法满足超大规模高并发需求。**应对思路**：优化 Candle 配置或使用量化机制，初期仅保障小规模验证可行。
