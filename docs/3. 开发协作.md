# Satori 开发协作

本文档记录当前仓库中会影响后续协作的开发约定。

## 1. 角色分工

- 前端由 Gemini 负责。
- 当前仓库中的 Codex 侧重点是后端、数据、测试、文档和 API 契约。

## 2. 数据提交边界

- `data/raw/` 不提交到仓库。
- `data/processed/imported/` 不提交到仓库。
- `data/index/` 不提交到仓库。
- `data/processed/cards.json` 可以提交，但只保留人工确认后的稳定词条。
- `tests/fixtures/` 保留最小且稳定的测试夹具，用于 smoke test 和 regression test。

这样做的原因：

- 原始语料和自动导入结果可重复生成，噪音大。
- 本地索引体积和格式可能频繁变化，不适合进入 Git。
- 稳定的人工语料和测试夹具需要被版本化，便于比较搜索质量变化。

## 3. CI

当前最小 CI 位于 `.github/workflows/ci.yml`，负责：

- `cargo fmt --all --check`
- `cargo test --workspace`

CI 目标是保证提交到 `main` 的格式和基础行为保持稳定。

## 4. CodeRabbit

仓库根目录的 `.coderabbit.yaml` 按当前项目形态做了最小配置：

- 忽略原始数据、导入产物、本地索引和演示 PDF。
- 对 `crates/api`、`crates/core`、`crates/indexer`、`tests` 和 `docs` 提供路径级评审说明。
- 让评审更关注 API 契约稳定性、搜索结果稳定性、导入可重复性和文档是否过时。
- 启用 CodeRabbit 的 code guidelines 读取能力，让仓库根目录的 `AGENTS.md` 参与评审上下文。

配置依据参考 CodeRabbit 官方文档中的仓库配置与路径级指引能力：

- https://docs.coderabbit.ai/

如果仓库的模块职责发生变化，需要同步更新 `.coderabbit.yaml`。

## 5. 当前协作重点

- 保持搜索 API 的路径和基础 JSON 结构稳定，避免给前端联调带来不必要破坏。
- 优先扩充稳定语料和回归样例，而不是过早引入复杂抽象。
- 在切换到 Embedding 和 LanceDB 前，先保住已有测试、CI 和契约文档的一致性。
